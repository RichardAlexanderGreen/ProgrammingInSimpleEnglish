The Principle of Simple

Problem-space complexity is conserved.

Complexity is conserved.
You can hide it.
You can abstract it.
You can defer dealing with it.
But, you cannot get rid of it. [1]
Some irreducible complexity is there in the problem space,
and eventually, that complexity must be represented in the solution space. [2]

The trick is to avoid adding unneeded complexity in the solution space.

While the complexity of the problem space is conserved,
a good tool does not add unneeded complexity.
What does this mean?

  • First of all, the tool should provide a simple way of representing the problem space.
    The notation should offer good fidelity.
    - It does not distort the basic facts of the problem space by recasting them into an awkward or esoteric representation.
  • Secondly, the different facets of the problem space
    should remain different facets in the solution space.
    For example: The domain model is not mixed with the presentation model.
  • Thirdly, the tool should be transparent.
    What does that mean?
    It means that the tool does not produce problems of its own.
    It should not add artifacts to the problem space descriptions.
    It should not obscure the facts of the problem space by translating them into an obscure notation.
  • Finally, simple recurring problems should have nearly automatic solutions.
    Once the problem statement is complete, the solution should be a push-button away.
    For example, if the presentation problem is to display a few related domain entities in an update mode, the display should be automatically produced along with the needed logic for updating the data store.

Hum avoids unneeded complexity and obscure notations.

Hum does have notations - but they are designed to be as natural as we could make them. The notations provide a regimen for organizing knowledge. But the regimen is as simple as we could make it.

The most esoteric notation in Hum is the mark-up notation used to advise the user interface actors. We have attempted to keep the mark-up notation minimal but adequate. Hopefully, this might be hidden from view some day when a WYSIWYG editor will obviate the need for people to know this notation.

[1] We deliberately ignore situations where people have created a self-inflicted complexity in the problem space and that complexity might be removed from the problem space by process re-engineering or policy reform. Consider, for example, the (USA) federal tax code.

[2] The principle is that the solution designers must acknowledge and represent whatever complexity exists in the problem space.
